> 💡 **核心原则**：高质量的代码不是“憋大招”写出来的，而是通过持续、高频、可验证的小步快跑交付出来的。

---

## 一、 开发节奏与周期 (Development Rhythm)

团队抛弃瀑布式开发，采用**敏捷迭代 (Agile Iteration)** 与 **持续集成 (CI)** 结合的节奏。

### 1. 双周迭代制 (Two-Week Sprints)
- **迭代长度**：固定为 2 周（10 个工作日）。
- **节奏拆解**：
  - **第 1 天**：迭代计划会 (Sprint Planning)。明确目标，拆解任务，确定每个任务的验收标准 (DoD)。
  - **第 2~8 天**：专注开发与高频代码提交。
  - **第 9 天**：代码冻结 (Code Freeze) 与回归测试。解决迭代内遗留的 Bug。
  - **第 10 天**：交付评审会 (Sprint Review) 与回顾会 (Retrospective)。部署可运行的中间版本。
- **例外处理**：节假日或紧急需求导致节奏变化时，需在迭代计划会同步调整并记录原因。

### 2. 代码提交节奏 (Commit Rhythm)
- **微提交 (Micro-Commits)**：避免一天结束时才提交一次包含几十个文件的“巨型 Commit”。
- **原子性**：一个 Commit 只做一件事（如封装一个 Button 组件，或新增一个 API 函数）。
- **建议**：以可审阅单元为粒度频繁提交；若确需大改动，必须拆分为多个可回滚的提交。

---

## 二、 目标拆分与任务粒度 (Goal Breakdown)

任何宏大的目标（如“实现移动端报工功能”）在进入开发前，必须被拆解为可在 **0.5 ~ 2 天**内完成的最小可交付单元 (Task)。

### 1. 垂直切片拆解法 (Vertical Slicing)
不要按技术层（“今天写所有 API，明天写所有 UI”）来拆分，要按**用户价值流**来拆分。
*错误拆法*：
- 任务1：写完后端的 10 个 API 接口。
- 任务2：写完所有的前端 Vue 页面。
*正确拆法 (垂直切片)*：
- 任务1：实现“扫码解析施工单信息”的全链路（从 UI 到 API 再到状态机）。
- 任务2：实现“提交完工数量并校验”的全链路。

### 2. 粒度红线
- 任何估算超过 **3 天 (24小时)** 的单个开发任务，**必须被强行打回重拆**。
- 如果一个任务描述里包含“和”、“并且”（如：“实现登录和注册功能”），说明它还不够小，必须拆成两个任务。

---

## 三、 验收标准与完成定义 (Definition of Done - DoD)

**“写完了”不等于“完成了”**。在本项目中，一个任务只有满足以下**全部条件**，才能在看板上拖入 `Done` 列：

### 1. 代码质量门禁 (Code Quality Gate)
- [ ] 代码通过本地 ESLint 与 Prettier 检查，**无任何 Warning**。
- [ ] 核心逻辑或工具函数必须配有单元测试，且**测试覆盖率达标**。
- [ ] 代码在本地构建无任何 TS 类型报错。
- [ ] 上述命令以仓库脚本为准（如 `pnpm lint` / `pnpm test` / `pnpm build`），若脚本缺失须先补齐。

### 2. 跨端回归验证 (Cross-Platform Verification)
由于是跨平台项目，开发者不能只在 Web 浏览器里测一下就认为完成。
- [ ] **Web 端**：在 Chrome/Edge 最新版中功能正常，响应式断点（缩放浏览器窗口）无布局崩塌。
- [ ] **桌面端 (Tauri)**：如果在本地启动 `apps/desktop`，相关功能（特别是涉及本地文件、原生窗口的功能）必须测试通过。
- [ ] **移动端体验**：如果在浏览器开启移动端模拟器（或本地跑 Capacitor），按钮触摸区域是否足够大？虚拟键盘弹起时是否遮挡输入框？

### 3. Code Review (代码审查) 与合并
- [ ] 开发者在 GitLab/GitHub 上发起 Pull Request (PR) / Merge Request (MR)。
- [ ] PR 标题符合 [Conventional Commits] 规范。
- [ ] PR 必须经过至少 **1 名核心开发者的 Approve**。
- [ ] 核心开发者名单以 `CODEOWNERS` 或技术负责人维护的权限表为准。
- [ ] CI 管道（自动化 Lint 与测试）必须显示为 **绿灯 (Passed)**。

### 4. 交付与文档 (Delivery & Docs)
- [ ] 如果修改了公共组件库 (`packages/ui-shared`)，必须在对应的组件示例文档（如 Storybook/VitePress）中同步更新用法。
- [ ] 如果引入了新的环境变量（如新的 API 密钥），必须更新工程根目录的 `.env.example`，并在群里通知团队。

---

> 💡 **总结**：我们不相信开发者的“我测过了，没问题”。我们只相信自动化测试的绿灯、严格的 Code Review 记录，以及最终打包出来的多端可执行文件。
