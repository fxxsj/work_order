# 多平台客户端规划 - 印刷施工单跟踪系统

> 更新时间：2026-02-22  
> 规划版本：v0.2  
> 适用系统版本：v3.0.0+（Django/DRF 后端 + 前端 legacy(Vue 2.7) 与 vNext(Vue 3) 并行）  
> 目标：交付 Web / macOS / Windows / Android 客户端，并最大化代码复用、可维护性与交付效率

## 1. 背景与现状

### 1.1 当前系统形态（仓库现状）

- **后端**：Django 4.2 + DRF + TokenAuthentication + Channels(WebSocket) + drf-spectacular(OpenAPI)
- **前端**：Vue 2.7 + Element UI + Vue CLI（SPA）
- **通信**：REST API（分页/过滤）+ WebSocket（通知）

### 1.3 当前实现进展（截至 2026-02-22）

仓库已落地“Web 业务代码复用 + 多端壳”最小闭环：

- Web vNext：`apps/web/`（Vue 3 + Vite + TypeScript + Pinia + Element Plus）
- 桌面壳：`apps/desktop/`（Tauri，macOS/Windows）
- Android 壳：`apps/mobile/`（Capacitor）
- 端侧运行时配置：登录页「服务器设置」（`API Base URL / WS Base URL`）
- OpenAPI 导出脚本：`scripts/openapi/export-backend-openapi.sh`
- SDK 骨架：`packages/sdk/`（基于 OpenAPI 生成 TS 类型/SDK）

### 1.2 多端诉求拆解

目标平台与最小能力边界：

- **Web**：全功能管理端（现有能力延续 + 长期可演进）
- **macOS / Windows**：桌面端（常见诉求：通知、自动更新、离线缓存、文件/打印能力、局域网部署适配）
- **Android**：移动端（常见诉求：触控友好、扫码/拍照/文件上传、弱网可用、通知）

关键原则：

- **一套业务逻辑，多端外壳**（优先），避免“每端重写一套”
- **先交付可用，再逐步变好**（分阶段），避免大爆炸重构导致停摆
- **API 契约优先**（OpenAPI/SDK 自动生成），减少多端重复集成成本

## 2. 技术路线选型

### 2.1 约束与决策点

必须回答的工程问题（会直接影响后续返工成本）：

1. **是否接受先“壳化”现有前端**（Vue 2.7）快速出桌面/Android，再迁移到 Vue 3？
2. **桌面端是否需要深度系统能力**（打印、串口/USB、托盘、开机启动、文件系统权限、自动更新）？
3. **Android 是否要“强原生体验”**（高帧率复杂动画/大量离线/深度硬件能力）还是以业务效率为主？

本规划默认目标为：**企业管理系统**（表单+列表+流程+通知），优先交付效率与可维护性。

### 2.2 推荐路线（主推）

推荐采用“单前端代码 + 多端壳”的组合：

- **Web**：Vue 3 + Vite + TypeScript + Pinia（UI：Element Plus 或 Ant Design Vue）
- **桌面（macOS/Windows）**：Tauri（加载同一套 Web 前端）
- **Android**：Capacitor（加载同一套 Web 前端，可按需补插件能力）
- **SDK**：基于 OpenAPI 生成 TypeScript 客户端（可选：再生成 Kotlin/Dart）

选择理由（面向本项目）：

- 业务形态适合 Web UI 复用（表格、筛选、弹窗、流程）
- Tauri/Capacitor 能快速把“已验证的 Web 功能”变成可安装应用
- 用 OpenAPI 生成 SDK，可以显著降低多端“手写请求 + 类型不一致”成本

### 2.3 备选路线（何时选）

1) **Flutter（桌面 + 移动同构）**  
适合：需要强离线、复杂交互、强原生体验，并且可接受更高的重写成本。  
代价：现有 Vue 代码复用较少，API/状态/组件需重建。

2) **Electron（桌面）**  
适合：大量 Node 生态依赖或复杂桌面能力，且包体/内存不是核心约束。  
代价：更重、更新链路与安全面更复杂；对“管理系统”常被 Tauri 替代。

## 3. 目标架构（vNext）

### 3.1 推荐目录形态（分阶段演进）

短期（不打断现有 `frontend/`）：

- `frontend/`：现有 Vue 2.7（继续维护）
- `apps/web/`：新 Web（Vue 3/Vite）逐步替换
- `apps/desktop/`：Tauri 容器（指向 `apps/web` 构建产物）
- `apps/mobile/`：Capacitor 容器（指向 `apps/web` 构建产物）
- `packages/sdk/`：OpenAPI 生成的 TypeScript SDK（Web/桌面/移动共享）
- `packages/shared/`：共享工具、权限/菜单模型、日期/货币/校验等纯逻辑

中期（稳定后再收敛为 monorepo）：

- 使用 `pnpm workspace` 或 `npm workspaces` 管理多包依赖
- CI 统一在根目录构建与发版

### 3.2 关键跨端能力设计

1) **环境配置（必须统一）**
- `API_BASE_URL`：桌面/安卓需要显式配置（生产/测试/局域网）
- `WS_BASE_URL`：通知通道（同上）
- “服务器选择”能力：Android/桌面提供 UI 配置（例如保存多个服务器地址）

2) **认证与 Token 存储**
- 保持现有 `Token` 认证可快速复用
- 桌面/安卓：Token 存储使用安全存储（Keychain/Keystore）而不是 localStorage
- WebSocket：避免长期 token 直接出现在 URL（若短期仍用 query token，需在反向代理/日志侧避免记录敏感参数）

3) **离线与弱网（按需分级）**
- L0：无离线，仅错误提示与重试
- L1：列表缓存（最近 N 天、最近 N 个施工单/任务）
- L2：可离线创建草稿（本地队列，在线后补提交）

4) **文件与打印**
- Web：浏览器能力为主（导出、下载、打印）
- 桌面：通过 Tauri 插件扩展（本地文件选择、打印调用、托盘等）
- Android：通过 Capacitor 插件扩展（相机、相册、文件、扫码）

## 4. 分阶段实施路线（建议）

> 目标是“每阶段都有可验收产物”，并且可随时停止在某一阶段仍保持系统可用。

### Phase 0：契约与基础设施（1–3 天）

交付物：
- OpenAPI schema 可稳定导出
- 生成 `TypeScript SDK`（统一请求封装、类型）
- 前端支持配置 `API_BASE_URL / WS_BASE_URL`（为桌面/安卓做准备）

验收标准：
- SDK 能跑通：登录、拉列表、更新、关键动作接口
- WebSocket 通知在开发环境可连通（含重连/心跳）

### Phase 1：快速多端 MVP（1–2 周）

目标：不等大重构，先把“可安装客户端”跑起来。

交付物：
- **桌面 MVP（macOS/Windows）**：Tauri 包壳，能登录、浏览/操作核心页面、通知可用
- **Android MVP**：Capacitor 包壳，能登录、浏览/操作核心页面、基础文件上传可用
- 发布与更新策略草案（内测渠道）

验收标准：
- 端到端业务链路可用（至少：登录→施工单列表→详情→任务操作→通知）
- 端侧可配置服务器地址

### Phase 2：Web 前端现代化（2–6 周，允许并行）

目标：从 Vue 2.7 迁移到 Vue 3/Vite/TS，提升长期可维护性。

实施策略（推荐“绞杀者模式”）：
- 新功能与新页面优先在 `apps/web`（Vue 3）开发
- 旧页面逐步迁移，按模块替换
- 迁移完成后再下线旧 `frontend/`

验收标准：
- 关键模块迁移完成（至少：登录/权限/菜单、施工单、任务、通知、基础字典）
- 单元测试/最小 e2e 可跑（至少覆盖登录与核心链路）

#### 当前落地状态（仓库内）

- 已新增 Web vNext 骨架：`apps/web/`（Vue 3 + Vite + TypeScript），已包含最小登录/路由守卫/运行时 API 配置入口。
- 现有 `frontend/`（Vue 2.7）继续作为稳定版本运行，迁移采用“逐步替换”。
- 已在 Web vNext 迁移/实现的模块（截至 2026-02-22）：
  - 施工单：列表 / 详情 / 审核 / 状态更新 / 创建（基础字段）
  - 任务：列表、操作员中心（认领/更新数量/完成）
  - 通知：列表、已读、WebSocket 连接（基础）
  - 基础数据：客户、产品、部门（含树形/工序配置）、工序（含批量启停）
  - 物料：物料、供应商、物料-供应商关联
  - 采购：采购单（创建/提交/批准/拒绝/下单/分批收货/取消）、收货质检/入库/退货中心
  - 库存：成品库存列表 + 调整 + 基础编辑、入库单、出库单、发货单（发货/签收/拒收）、质量检验（含完成检验）
  - 销售：销售订单（创建/提交/审核/拒绝/开始生产）
  - 财务：发票（创建/提交/审核/作废）、收款记录（创建/删除）
  - 资产：图稿（确认/新版本）、刀模（确认）、烫金版（确认）、压凸版（确认）

### Phase 3：体验与原生能力增强（按需滚动）

桌面优先项：
- 自动更新、托盘、开机启动（如需要）
- 打印/导出体验优化
- 本地文件与附件管理增强

Android 优先项：
- 扫码/拍照上传
- 弱网优化（请求重试、离线草稿）
- 推送通知（如需 FCM）

## 5. 关键风险与对策

### 5.1 Vue 2 → Vue 3 的迁移成本

风险：Element UI 与 Vue 3 不兼容（需切换 Element Plus 或替代 UI）。  
对策：Phase 1 先“壳化”现有前端出客户端，Phase 2 再迁移 UI/框架。

### 5.2 Token + WebSocket token in query 的安全性

风险：URL 可能被代理/日志记录，泄露 token。  
对策（从低到高）：
- 低：Nginx/应用日志过滤 query 参数；缩短 token 有效期；设备丢失可立即失效 token
- 中：新增“WS 票据”接口（短时票据换取 WebSocket 连接）
- 高：升级到 JWT + 短期 access token + refresh token（并对 WS 做票据/子协议）

### 5.3 多端发布与签名

风险：macOS 签名、公证；Windows 签名；Android 签名与渠道差异。  
对策：优先内测渠道跑通（不强制签名），正式发布再补齐签名与自动更新。

## 6. 工作拆分（Backlog 提纲）

### 6.1 后端（支撑多端）
- CORS/CSRF/AllowedHosts 生产化配置（包含桌面/安卓 origin）
- OpenAPI schema 固化与版本管理（建议引入 `/api/schema/` 的导出脚本）
- 可选：WS 票据接口（短时有效）

### 6.2 前端（共享能力）
- 统一配置层（读取环境变量 + 端侧存储 + UI 设置页）
- 统一 SDK（OpenAPI 生成 + 拦截器 + 错误模型）
- 统一认证、权限、菜单模型（减少多端重复）
- 通知中心：WebSocket 连接生命周期与后台/前台行为（桌面/安卓差异处理）

### 6.3 桌面（Tauri）
- 容器工程（窗口、菜单、自动更新、系统通知）
- 文件选择/导出/打印（按需）

### 6.4 Android（Capacitor）
- 容器工程（WebView、权限、文件/相机）
- 本地存储与安全存储
- 推送通知（按需）

## 7. 验收清单（各端一致）

核心功能验收（建议用同一套用例）：

- 登录/登出、token 过期处理
- 施工单：列表/筛选/详情/创建/编辑/审核（按权限）
- 任务：认领/开始/完成/记录数量（按权限）
- 通知：实时到达、已读、未读数、跨会话恢复
- 异常：断网、重试、并发冲突（409）提示

## 8. 里程碑（可调整）

- M0：Phase 0 完成（契约 + SDK + 可配置）
- M1：桌面 MVP 内测包（macOS/Windows）
- M2：Android MVP 内测包
- M3：Web vNext（Vue 3/Vite/TS）核心模块上线
- M4：体验增强（离线/推送/自动更新/打印等）

## 附录 A：建议的“先做什么”

如果目标是尽快看到多端成果，优先顺序建议为：

1. **把 API_BASE_URL/WS_BASE_URL 改为可配置**
2. **OpenAPI 生成 TS SDK（让多端同一套请求/类型）**
3. **先出桌面壳（Tauri）与 Android 壳（Capacitor）**
4. **再做 Vue 3/Vite 迁移（长期维护性）**
